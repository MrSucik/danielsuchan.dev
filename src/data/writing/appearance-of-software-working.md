# The appearance of software working is not software working

On the Q2 2025 Palantir earnings call, Alex Karp dropped a line that has stuck with me: *"The appearance of software working is not software working."* He was talking about what he called AI slop — tools that demo well, get praised on stage, and quietly fail in the field. The line works because it splits two things that the LLM era has glued back together: how a system looks while it runs, and what it actually does.

I run into this every week. I build [Dzarvis](https://dzarvis.com) — a multi-agent assistant on Claude — and an algorithmic trading fund. Both are environments where the cost of "looks fine, isn't fine" is asymmetrically high. After a year of shipping production agents, I think Karp's observation is the single most important framing for anyone building with LLMs right now. The tools have outpaced the checks. We are deploying systems whose surface behavior is much smoother than their substrate. The cleanup will be expensive.

## Where the gap shows up

Pick any agent loop in production and you will find at least three of these:

- The agent "completes" the task because the orchestrator sees a non-error return code. The result is wrong.
- A tool call looks structured (JSON, schema-conforming, the SDK is happy) but the values inside violate a domain invariant nothing was checking.
- The CI pipeline is green: types compile, lint passes, all tests run. The migration drops a column the dashboard reads.
- The output reads fluent. The reader nods. Three days later someone realises the citation was hallucinated.
- Telemetry shows latency improved. It improved because half the requests now silently fall through to a fallback that no longer reaches the database.
- A code-review bot gives the PR a thumbs up. It read the diff, not the system the diff sits inside.

In each case there is a green light. Each green light is generated by a process that measures *the appearance of working*. None of them measure working.

This is not new. The field has known about it for decades — it's why integration tests exist, why we have things like "tests should fail before you make them pass," why senior engineers ask "what evidence do you have." What is new is the ratio. With AI in the loop, the cost of producing a green light has collapsed. The cost of producing actual confidence has not.

## Why LLMs reward appearance

Models are trained on examples where the *visible signal* matched the *underlying behavior*. A correct answer in pre-training data was an answer that looked correct in text. A passing test was a test someone wrote to be passed. The training signal could not, in most cases, distinguish between "this code solves the problem" and "this code is the kind of code that usually solves problems like this." So the model learned to produce things that pattern-match to working code, working answers, working analyses.

Karp's framing is sharper than the standard "LLMs hallucinate" complaint because it doesn't single out the model. It points at the whole stack. A model produces plausible code. A bot reviews it for plausibility. CI runs the tests the model wrote, which exercise the paths the model considered. A human reads a diff that compresses the change to a shape that looks reasonable. A dashboard reports green. Five things in a row optimised for "looks like working." None of them optimised for *working*.

In my own work, the moment I started taking this seriously was when an autosales pipeline I'd built started producing emails that read perfectly: subject line on point, salutation correct, three crisp bullets, signature aligned with the sender's voice. The metric "draft generated" was at 100%. The metric "lead actually exists at this company in this role" was around 70%. The model was filling in fields with the right shape. We had built a beautiful machine for producing *plausible* outreach.

## What "working" actually requires

The fix is not glamorous and it is not new. It is the boring engineering culture the AI bubble has been quietly eroding. A working system is one that has been tested against the substrate it claims to operate on, with checks that fail noisily and do not lie when they fail.

Concretely, the moves that have helped me most:

**Don't mock the boundary.** Integration tests against a real Postgres beat unit tests against a mock library every time. Mocks lie about what the upstream actually does, and LLMs are very good at making code that satisfies a mock without satisfying reality. Tests should hit a database, an API, a queue, a sandbox — the actual thing.

**Verify outputs before they cross trust boundaries.** Anything an LLM produces that touches the database, the inbox, or a customer's account gets parsed, validated, and pattern-checked before it is allowed through. Zod-parse the structured output even if the SDK swears it is structured. URL-parse and allowlist-check anything that becomes a `fetch`. Treat the model as an *untrusted source*, because that's what it is.

**Pair every claim with evidence.** When an agent says "I created the calendar," it shows you the calendar ID and the API response. When CI says "passes," it shows you the test count and the assertion strings. When a code reviewer says "looks good," it points at the line where the invariant is now enforced. Plain-English claims without artifacts are how a system slides from working to "working."

**Read the code path that ran, not the code path you wrote.** The most expensive bug I've shipped this year was a feature that compiled, deployed, and ran — but ran a stale generated bundle from before my last edit. The dashboard said green. The behavior was on the previous version. The fix was to make the code path observable end-to-end so the green light could not lie.

**Telemetry is truth, not vibes.** If you are running an agent in production, you should be able to answer: how often does it succeed, how often does it silently degrade, what does the failure distribution look like, what fraction of the "successes" hit the cheap fallback. If you can't answer those, the green light is decorative.

## What this changes about how I hire and build

The bottleneck on AI-assisted engineering is no longer producing code. It is producing *justified confidence* in code. That changes who is valuable on a team. The engineers I want now are the ones who instinctively ask "how do we know" before they ask "how do we ship." They are the ones who write a five-line skeptical test before accepting a 200-line agent diff. They are the ones who read the telemetry without being asked.

It also changes how I scope work. A feature is not done when it compiles. It is done when there is a pattern of evidence — telemetry, tests against the real substrate, a verifiable artifact for every behavior the spec claims — that the substrate is doing what the surface says.

Karp's line, restated for builders: *demo confidence is now free. Real confidence is now the entire job.*
